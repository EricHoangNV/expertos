The builder is the **single enforcement point** for voice-on-top-of-facts — M2.4's voice-vs-facts separation tests should assert against its output (and against a live LLM in the eval harness), not re-implement the rule. (2) `VoiceService.retrieveVoice(user, voiceQuery)` keys on `expertId` + `language`; it returns `{ profile: null, examples: [] }` when the expert has no *published* profile in that language — the caller must fall back to a neutral voice (call `buildAnswerPrompt` without `voice`), facts still enforced. **Voice-example embeddings must be written with the same model as the query** (both from `createDefaultEmbeddingProvider`) — they aren't populated yet (no admin authoring/seed for voice examples); when seeding voice examples for M2.2 demo, embed via that factory or cosine match is meaningless. The `voice_examples.embedding` raw-SQL cosine query is **not** exercised against real pgvector here (same as `PgVectorStore`) — add it to the M11 Testcontainers integration pass.
- **Eval harness seam (M1.3):** `evaluateRetrieval(RETRIEVAL_GOLDEN_SET, opts)` from `@expertos/ai` is the RAG eval entry point. It is **deterministic and offline** by design — pass `{ embedder }` to swap the default `HashingEmbeddingProvider` (lexical only) for the real multilingual model in an out-of-band run to get the *semantic* VI/cross-lingual numbers (the offline run only guards tokenization/normalization/fusion). **Important nuance:** with the lexical hashing embedder a tiny corpus trivially returns every doc (all chunks fit in `topK`), so meaningful recall/MRR needs `topK` < corpus size — the seed golden set has 12 docs for this reason. The cross-lingual *miss* is intentional and asserted (`harness.test.ts`): the lexical model cannot bridge EN↔VI; only the real model can. Extend this for M2.4 (add voice-fidelity / voice-on≈voice-off assertions) and M4 (citation-resolvability) rather than starting a new harness.
- **NFC normalization is load-bearing for VI (M1.3):** all indexed text (chunk content) and all query text MUST be NFC-normalized or Vietnamese recall silently collapses (NFD combining marks shatter tokens). This is enforced at three boundaries — `@expertos/ai` `text.ts` `tokenize`/`normalizeText` (embedder + chunker + eval share one tokenizer), and `@expertos/shared` `retrievalQuerySchema` `.transform(normalizeText)` for query text (covers the Postgres keyword path, which does NOT normalize). When the real OpenAI embedder driver lands, keep NFC normalization in its tokenization path too. See LEARNINGS.
- **Retrieval seam (M1.2):** Call `RetrievalService.retrieve(user, query)` (`apps/api/src/retrieval`) — `query` is the shared `retrievalQuerySchema` (`text`, `topK` default 8, `filters` with `status` default `published`). It returns `RetrievedChunk[]` (`chunkId`, `documentVersionId`, `content`, fused `score`, optional `vectorScore`/`keywordScore`) — exactly what the M4 citation builder needs (`documentVersionId` provenance is already carried). The `VectorStore` interface is now `retrieve(RetrievalRequest)`; swap pgvector for Vertex/Qdrant behind it. Pure ranking lives in `@expertos/ai` `fuseHybrid` (RRF) — extend there, not in the driver. **The query embedder MUST stay the same model as ingestion** (both wired from `createDefaultEmbeddingProvider`); when the real OpenAI driver lands, change that one factory so both sides move together, else query/chunk vectors stop being comparable.
